# RFC-001: Core Architecture Foundations

**Status:** Retrospective traceability record — pending Architecture Owner confirmation  
**Scope:** Observation, Browser Runtime, Session Manager, Validation, and Execution Kernel

## Purpose

This RFC records the architectural alternatives and trade-offs underlying ADR-001 through ADR-005. It is retained as the required pre-decision traceability artifact for the approved base architecture. It does not alter those decisions or authorize implementation.

## Problem Statement

WIP must inspect and reconstruct web experiences without asking a reasoning model to infer browser facts that can be measured directly. The system also needs explicit execution semantics, session isolation, and deterministic validation.

## Options Considered

### Option A — Direct Coordinator-to-Browser Automation

The Coordinator would call a browser automation framework directly and reason over raw DOM, CSS, and screenshots.

**Rejected because:** it couples probabilistic reasoning to browser-specific APIs, permits layout speculation, and distributes execution semantics across runtimes.

### Option B — Browser Runtime with Observation Model and Graph

The Browser Runtime exposes bounded deterministic capabilities. Measurements are represented first as an Observation Model and then as an Observation Graph. The Coordinator reasons only over those facts.

**Benefits:** preserves observation/reasoning separation, supports framework-agnostic consumers, and enables structural queries and deterministic comparison.

**Costs:** requires explicit schemas, graph construction, and a capable browser instrumentation layer.

### Option C — Browser Runtime Plus an Execution Kernel

An Execution Kernel mediates all Coordinator actions and owns missions, permissions, scheduling, retries, cancellation, checkpoints, and event ordering. A Session Manager isolates browser state. Validation runs in an isolated harness after reconstruction.

**Benefits:** centralizes execution policy, preserves deterministic runtime boundaries, supports reproducibility, and makes validation mandatory.

**Costs:** adds protocol and lifecycle complexity and requires formal contracts before implementation.

## Recommendation

Adopt Option C, with the Observation Model and Observation Graph from Option B. Keep the Browser Runtime limited to deterministic navigation, viewport, interaction, and observation capabilities. Keep reasoning, orchestration, and validation policy outside the runtime.

## Decision Mapping

| Decision | ADR | Resulting contract |
| --- | --- | --- |
| Observation Model and Graph | ADR-001 | Facts are immutable and relationships are explicit and queryable. |
| Browser Runtime boundary | ADR-002 | Browser control is exposed only through bounded deterministic capabilities. |
| Session isolation | ADR-003 | Runtime lifecycle and state are isolated and managed explicitly. |
| Validation loop | ADR-004 | Reconstruction is verified through structural and visual comparison. |
| Execution semantics | ADR-005 | Coordinator actions pass through the Execution Kernel. |

## Approval Boundary

This RFC establishes traceability for the existing ADRs. It does not lock Missions 3–8, change any interface, select a browser automation implementation, or authorize Mission 9 coding. Those actions require the existing design-review and approval gates.
